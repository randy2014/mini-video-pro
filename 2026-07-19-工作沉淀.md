## 二、关键坑 & 根因（重点，以后少走弯路）

### ⚠️ 坑 1：真·源码位置陷阱（最容易白改，已踩两次）
- **git 仓库根**：`/workspace/video-entitlement-server/`（remote = `randy2014/mini-video-pro`）。
- **前端（React）真实路径**：仓库根下直接挂 `video-entitlement-admin/src/...`，改这里就对了。
- **后端（Spring Boot）真实路径是「双层嵌套」** ⚠️：
  - 真正的 Spring 模块在 `/workspace/video-entitlement-server/video-entitlement-server/src/main/java/com/video/entitlement/module/platform/...`
  - 即仓库根里**还有一个同名 `video-entitlement-server/` 子目录**，源码在「第二层」里面；包名是 `com.video.entitlement.module`（不是 `com.example.video`）。
  - **血泪教训**：直接对 `video-entitlement-server/src/...`（第一层）改/查，会落到空壳/错误路径，提交后线上毫无变化、grep 还查不到。所有后端改动与核查都必须进到**第二层 `video-entitlement-server/` 里面**。
- ❌ `/workspace/video-entitlement-admin/` 是**另一个独立 git 仓库副本**，不参与部署，改它白改。
- ⚠️ 仓库根下还有**游离的未跟踪目录** `src/`、`target/`、`docker/`（构建残留/错误路径产物，里面是空的或陈旧脚本），**不是源码、不要去改**，也别 `git add`（仓库原无 `.gitignore`，现已补，见「五、待留意」）。

### ⚠️ 坑 2：CI/CD 红线
- 部署**只能走 GitHub Actions**（push `main` 触发 `video-entitlement-server/.github/workflows/deploy.yml`）：构建前后端镜像 → 推 GHCR → SSH 拉镜像重启 `video-frontend` / `video-backend` 容器。
- **禁止本地构建后 scp/手动部署**；Flyway 禁止手动迁库；不要 `docker cp` 做持久化（用 `-v` 卷挂载）。
- 判断部署是否生效：看 `gh run list` 的 `Deploy to Server` run 是否 `success`，并 curl 线上 bundle 哈希是否变化。

### ⚠️ 坑 3：AppVersionController 路径 bug（预存 bug，本次才暴露）
- 旧代码类级 `@RequestMapping("/api/v1")` + 管理端方法 `@GetMapping("/admin/versions")` → 实际解析为 **`/api/v1/admin/versions`**。
- 但安全规则与前端都期望 **`/admin/api/v1/versions`**（admin 在前，与其余所有 admin 控制器一致）。
- 结果：管理端版本列表等接口**一直 404 → NoResourceFoundException（10500）**，从没真正可用过。本次改为每个方法写全路径 `/admin/api/v1/...`，客户端接口保留 `/api/v1/client/version`。

### ⚠️ 坑 4：Spring 上传默认 1MB 上限
- 默认 `spring.servlet.multipart.max-file-size=1MB`，APK 约 5–7MB → `MaxUploadSizeExceededException`。
- 已在 `application.yml` 设 `max-file-size: 50MB` / `max-request-size: 60MB`。
- ⚠️ **教训**：本次第一次加这段配置时编辑“看似成功”但提交后文件里没有（疑似编辑未落盘/被覆盖）。**提交前务必 `grep -n multipart application.yml` 确认存在**，否则线上仍是 1MB 默认。

### ⚠️ 坑 5：上传落盘必须给后端挂卷
- APK 由宿主机 `/data/video-apk` → 前端容器 `/usr/share/nginx/html/downloads:ro` 只读对外。
- 后端容器**原本没挂这个目录**，无法写上传文件。已在 `deploy.yml` 后端 `docker run` 增加：
  - `-v /data/video-apk:/data/video-apk`（RW）
  - `-e APP_APK_DIR=/data/video-apk -e APP_DOWNLOAD_BASE=http://43.161.222.78:8082/downloads`

### 坑 6：日期格式规范（统一约定）
- 所有列表/日志时间列统一 `dayjs(s).format('YYYY-MM-DD HH:mm')`；空值显示 `-`。
- 后端返回 ISO（`jackson` 已配 `write-dates-as-timestamps: false`），dayjs 可直接解析。

## 四、版本管理接口契约（新增）

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| GET | `/admin/api/v1/versions` | ADMIN | 版本列表（倒序） |
| POST | `/admin/api/v1/versions` | ADMIN | 发布：JSON `{versionName, versionCode, downloadUrl?, releaseNotes?, forceUpdate?}` |
| POST | `/admin/api/v1/versions/upload` | ADMIN | 发布：multipart `file`(可选)+`versionName`+`versionCode`+`releaseNotes?`+`forceUpdate?`；有文件则自动落盘生成下载地址 |
| PUT | `/admin/api/v1/versions/{id}` | ADMIN | 修改元数据（JSON） |
| POST | `/admin/api/v1/versions/{id}/upload` | ADMIN | 替换该版本的 APK 文件 |
| DELETE | `/admin/api/v1/versions/{id}` | ADMIN | 删除（记录+磁盘文件） |
| PUT | `/admin/api/v1/versions/{id}/status?status=ACTIVE\|INACTIVE` | ADMIN | 启用/停用 |
| GET | `/api/v1/client/version` | 公开 | Android 端拉取最新 ACTIVE 版本（`versionCode`/`downloadUrl`/`forceUpdate`） |

- 文件名生成：`video-entitlement-v{yyyyMMddHHmmss}.apk`；存储于 `${APP_APK_DIR:/data/video-apk}`，访问基址 `${APP_DOWNLOAD_BASE:http://43.161.222.78:8082/downloads}`。
- `versionCode` 须为正整数且唯一；`releaseNotes` 中文经 utf8mb4 + JSON UTF-8 写入，无乱码。
